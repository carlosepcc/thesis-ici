\subsection{Requisitos funcionales}
En ingeniería de software, los \ac{rf} definen un sistema o sus componentes; describen la función que un software debe realizar, ya sean cálculos, manipulación de datos, procesos de negocios, entre otros.
Ayudan además a capturar los comportamientos planificados para un sistema, este comportamiento puede ser expresado como una función, servicio o tarea que un software debe realizar \citep{pressman2005software}. A continuación, se exponen los diferentes \ac{rf} planteados por el usuario:
\begin{RF}
    \item Autenticar Usuario
    \item Asignar rol a usuario
    \\
    \item Listar denuncias %w
    \item Crear denuncia
    \item Modificar denuncia
    \item Eliminar denuncia
    \item Buscar denuncia
    %\item Exportar tabla de denuncias
    \item Exportar denuncia
    %\item Imprimir listado de denuncias
    \\
    \item Listar resoluciones decanales %w
    \item Crear resolución decanal %w
    \item Modificar resolución decanal
    \item Eliminar resolución decanal %w
    \item Exportar resolución decanal
    \\
    \item Listar comisiones %w
    \item Crear comisión %w
    \item Modificar comisión
    \item Eliminar comisión %w
    \item Buscar comisión %w
    \\
    \item Listar declaraciones
    \item Crear declaración
    \item Modificar declaración
    \item Eliminar declaración
    \item Buscar declaración
    % \item Exportar declaración
    %\item Exportar tabla de declaraciones
    \\
    \item Crear caso disciplinario
    \item Listar casos disciplinarios
    \item Modificar caso disciplinario
    \item Buscar caso disciplinario
    \\
    \item Exportar resolución de caso
    \item Exportar conclusión de la comisión
    \\
    \item Listar roles
    \item Crear rol
    \item Modificar rol
    \item Eliminar rol
    \item Buscar rol
    %\item Exportar tabla de usuarios
\end{RF}

\subsection{Requisitos no funcionales}
Los \ac{rnf}, como su nombre sugiere, son aquellos requerimientos que no se refieren directamente a las funciones específicas que proporciona el sistema, sino a las propiedades emergentes de este como la fiabilidad, el tiempo de respuesta y la capacidad de almacenamiento. De forma alternativa definen las restricciones del sistema \citep{sommerville2011software}.\\
Los requisitos no funcionales de la aplicación a desarrollar son:
\\
\noindent Usabilidad:
\begin{RNF}
    \item La \ac{gui} de la aplicación cliente debe ser intuitiva y fácil de usar por cualquier persona con edad laboral y con conocimientos básicos de trabajo con software, de acuerdo a estándares modernos de diseño de \ac{gui}.
\end{RNF}
Hardware y software:
\begin{RNF}[resume]
    \item El software debe poder funcionar en un servidor con 4GB de \ac{ram} y 10GB de almacenamiento libre.
    \item El cliente del sistema debe ser compatible con las últimas versiones estables de los principales navegadores web: Google Chrome, navegadores basados en \gls{chromium}, Mozilla Firefox y Safari.
    %TODO Seguridad?  \item El usuario debe poder acceder a las funcionalidades del sistema sólo después de estar autenticado.
    % \item El usuario debe poder acceder solamente a las funcionalidades permitidas de acuerdo a su rol en el sistema.
\end{RNF}
Diseño e implementación:
\begin{RNF}[resume]
    \item El sistema debe ser accesible a través de una aplicación web cliente.
    \item Se deben usar lenguajes de programación y herramientas com amplio soporte y estabilidad.
\end{RNF}
